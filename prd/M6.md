# M6 — Files, projects, system, faithful stubs (PRD)

> Status: ⬜ · Depends on: M1 (parallel with M2-M5) · Read `prd/README.md`. Tasks: `PRD.md` (M6-T*).

## Context / why

SmileBASIC programs save/load files and projects, read system variables, edit program
source at runtime (PRG* family), and interact with device features (mic, motion, wireless)
that we must reproduce *faithfully including their limitations*. This milestone also
finalizes multi-slot semantics (EXEC/USE/CALL/COMMON DEF). Storage must work on both
native (filesystem) and wasm (IndexedDB).

## Goal (Definition of Done)

- A storage layer (native FS / wasm IndexedDB) with an extdata-compatible layout.
- File commands (SAVE/LOAD/FILES/DELETE/RENAME/COPY/CHKFILE/PROJECT) per spec.
- System variables behave per spec (incl. faithful FREEMEM/HARDWARE models).
- PRG* source-editing commands work; mic/motion/MP behave with the right
  availability errors; DIALOG/FONTDEF/CLIPBOARD implemented.
- Multi-slot + COMMON DEF + EXEC/USE/CALL semantics complete.

## Reference sources
- Specs: `category` Files / Source-manip / Various input / Advanced control in
  `spec/instructions/`; `spec/reference/sysvars.yaml`, `errors.yaml` (36/37/38/43/44/45/46).
- Docs: `sb-docs/smilebasic-3/manual/` file/project topics.
- osb: `project.d` (Projects, Resource, FileName), `program.d` (Program/Slot, PRG* edits),
  `systemvariable.d`, `dialog.d`, `petitcomputer.d` (slots, EXEC/USE), `compiler.d`
  (MIC/MOTION/MP sysvars stubbed to 0).
- Disassembly: file format details + exact sysvar values where they matter.

## Architecture decisions
- `sb-core` defines a **`Storage` trait** (list/read/write/delete/rename); native and wasm
  implement it. Keep actual I/O out of `sb-core`. The extdata-compatible layout lets the
  oracle harvest interoperate (O-T3).
- "Faithful stub" means: reproduce the *observable behavior* of unavailable features
  (e.g. MIC without `XON MIC` → errnum 36; MP sysvars → fixed values), not a real
  implementation.

### M6-T6 — Multi-slot semantics
- **Files:** `crates/sb-core/src/vm.rs`.
- **Approach:** finalize 4 slots + shared COMMON DEF; EXEC (run another slot), USE (mark
  slot), CALL (call by name), cross-slot GOTO/GOSUB and variable scoping. → M1-T6
- **Acceptance:** programs spanning slots with COMMON DEF run; EXEC/USE/CALL behave per
  spec; CALLIDX correct in SPFUNC/BGFUNC callbacks.
- **Progress (CALL by name landed):** `Op::CallDynamic` is now implemented
  (`vm.rs::call_dynamic` + the shared `invoke_function` split out of `call_user`): `CALL
  "name" [,args] [OUT outs]` resolves a `DEF` instruction by a **runtime** name string
  (ASCII case-folded, no type suffix) and invokes it exactly like a literal `CallUser` —
  a non-string name → Type mismatch (8), an unknown instruction → Undefined function (16),
  both matching the hw_verified `call.yaml`. `CALL`/`COMMON` are folded into the
  conformance gate (`IN_SCOPE_CONTROL`): `call.yaml` (`calls_by_name`,
  `undefined_instruction`) + `common.yaml` (`common_def_same_slot`) now replay green. 7 new
  VM e2e tests (by-name, case-insensitive, string-var target, value args, OUT results,
  undefined → 16, non-string → 8). **Still `[ ]` — remaining multi-slot work:** EXEC (run
  another slot) + USE (mark a slot executable) are still `VmError::Unsupported` — both need
  a multi-program execution model (the VM holds one `Program`/`globals` for the running
  slot; a faithful EXEC context-switch + cross-slot COMMON DEF dispatch + per-slot variable
  scoping require threading a "current slot" execution context through `run`/`cell`/
  `var_suffix`/function lookup, plus a host hook to load compiled programs into slots 1-3).
  EXEC/USE runtime effect + errnums are all hypothesis/oracle-pending (their handlers aren't
  body-readable — `exec.yaml`/`use.yaml`), so faking a no-op switch would be wrong; left
  Unsupported until the multi-program model lands. CALLIDX is wired (`sysvars`, reads 0
  outside any callback) but the SPFUNC/BGFUNC per-frame callback **dispatch** that would set
  it is the M3-T6/frame-loop subsystem (no headless caller invokes the registered callbacks
  yet) — queued.
- **Progress (USE/EXEC error model + slot-executable flag landed, 2026-06-23):** harvested the
  full USE/EXEC validation behavior from the oracle and implemented it (`vm.rs::do_use`/
  `do_exec` + `parse_prg_operand`). **hw_verified:** USE numeric — slot outside 0..3 → errnum
  10, the running slot (`USE 0`) → 4, a valid non-running slot → marks it executable
  (`slot_used` flag + `Vm::slot_used` getter); USE string — `PRG0:` (running) → 4, unknown
  resource / PRG index past family (`FOO:`/`PRG4:`/`PRG5:`) / empty name → 4 (NOT 10 like the
  SAVE resolver), valid slot + missing file → 46, valid slot + existing file → marks it. EXEC —
  numeric slot out of range → 10, an empty non-running slot → Syntax error 3, string unknown
  resource → 4, missing file → 46. Specs `use.yaml`/`exec.yaml` corrected (the old hypotheses
  had range → 4, which is wrong) + folded into the conformance gate (`IN_SCOPE_MULTISLOT`); 13
  new vm.rs e2e tests. **Still `[ ]` — the multi-program CONTROL TRANSFER is the remaining
  work and is left `VmError::Unsupported` (not faked):** loading a compiled program into a
  slot (no host compile hook / slot-program registry yet — `EXEC`/`USE "PRGn:file"` success
  paths), the actual EXEC program switch + the no-return-vs-`END`-returns-across-slots rule,
  per-slot vs shared (COMMON) variable scoping, and cross-slot DEF/label resolution from a
  USE'd slot. These need a multi-program execution model + a ≥2-slot oracle harness to ground
  (the handlers are parser special forms with no body-readable disassembly) — queued in
  `HARVEST_QUEUE.md`.
- **Progress (cross-slot COMMON DEF dispatch landed, 2026-06-23):** the first piece of the
  multi-program execution model — "programs spanning slots with COMMON DEF run." The VM now
  holds a per-slot program registry (`slot_programs: [Option<LoadedSlot>; 4]`, each slot's own
  `Program` + `globals`) + an `active_slot`; a host/test seeds slots 1-3 via the new
  `Vm::load_slot_program(slot, Program)`. Call resolution (`resolve_user_function` behind both
  `Op::CallUser` and `Op::CallDynamic`) now falls back from the active program to a **`COMMON
  DEF` in a `USE`'d, loaded slot**; the cross-slot invoke swaps that slot's program/globals into
  the active context (`activate_slot`), tags the frame with its `caller_slot`, and `return_func`
  swaps the caller's context back before resuming. `ERRPRG` now reports `active_slot`. This is
  the **documented** model (common.yaml: COMMON makes a DEF callable from another slot after
  USE; call/arg/OUT/return semantics are those of DEF): `CALL "name"` after `USE n` runs the
  COMMON DEF with value args / OUT results / return value; an un-USE'd slot or a non-COMMON DEF
  → Undefined function (16). 6 new vm.rs e2e tests (cross-slot call after USE, args+OUT, return
  value, no-USE→16, non-COMMON→16, caller-context restored). **Still `[ ]`:** EXEC whole-program
  control transfer (form 1 string LOAD + form 2 numeric switch, the END-returns-across-slots
  rule) is still `VmError::Unsupported`; and whether a *free* (non-param/local) variable inside a
  COMMON DEF binds to the defining slot's globals (what we implement) or the caller's is
  oracle-pending (needs a 2-slot oracle sharing a same-named global) — both queued in
  `HARVEST_QUEUE.md`.
- **Progress (EXEC numeric loaded-slot control transfer landed, 2026-06-23):** the second
  piece of the multi-program execution model — `EXEC n` now *transfers control* to a loaded,
  non-running slot (`vm.rs::exec_transfer`). This is the **documented** model (exec.yaml:
  "Executes a program in a different SLOT"): `exec_transfer` swaps the target slot's
  program/globals into the active context (reusing `activate_slot`), resets the PC to the
  top, and — because it is impossible to return to the previous program — discards the
  caller's whole execution state (`DEF` frames, `GOSUB` stack, operand stack, `DATA` cursor);
  the EXEC'd program then runs against its own globals and, when it ends, the run ends.
  `current_slot`/`active_slot` both move to the target so `SLOT()`/`ERRPRG` report it. An
  *empty* non-running slot stays Syntax error (3, hw_verified); out-of-range stays 10. A
  host/test seeds slots 1-3 via `Vm::load_slot_program`. 5 new vm.rs e2e tests (transfer +
  abandoned tail, own-globals scoping, empty-slot→3, GOSUB-state discarded, ERRPRG=target
  slot). **Still `[ ]`:** the string FORM 1 file LOAD (needs a compile-from-storage hook
  threading the builtin registry into the VM), the running-slot RESTART (`EXEC 0`), the
  nested END-returns-across-slots rule, and the per-slot vs shared variable-scoping
  confirmation all remain `VmError::Unsupported`/oracle-pending — they need a ≥2-slot oracle
  harness, NOT a single injected program (queued, `HARVEST_QUEUE.md`). CALLIDX in
  SPFUNC/BGFUNC callbacks still rides the M3-T6 frame-loop dispatch.
- **Progress (EXEC/USE string-form `"PRGn:file"` file LOAD landed, 2026-06-23):** the third
  piece of the multi-program execution model — loading a program into a slot *from storage*.
  The blocker noted above ("needs a compile-from-storage hook threading the builtin registry
  into the VM") turned out not to need an external host hook: sb-core already has the whole
  `parse`→`compile_with(&StdBuiltins)` pipeline, so the new `vm.rs::compile_slot_file` reads
  the TXT body from the VM's `Storage` (UTF-8, like `LOAD "TXT:"`) and lowers it in-VM. Wired
  into both string forms: `EXEC "PRGn:file"` into a non-running `PRGn:` slot loads the file's
  program into that slot (`load_slot_program`) and **transfers control** to it (`exec_transfer`)
  — documented form 1, "Loads and executes a program"; `USE "PRGn:file"` loads the program and
  marks the slot executable, so its `COMMON DEF`s resolve cross-slot via `CALL "name"` (the
  dispatch landed earlier). The hw_verified error model is unchanged (missing file → 46,
  bad/empty resource → 4, running slot → 4 for USE). 7 new vm.rs e2e tests (EXEC load+transfer,
  own-globals scoping, missing-file 46, running-slot still deferred; USE load+call, args+OUT,
  missing-file 46) + the prior `exec_valid_target…unsupported` test refit to assert the real
  SAVE→EXEC transfer. **Still `[ ]`:** the running-slot RESTART (`EXEC 0`), the bare-name
  (no `PRGn:`) default-slot load, the nested END-returns-across-slots rule, and the per-slot vs
  shared variable-scoping confirmation stay `VmError::Unsupported`/oracle-pending (need a
  ≥2-slot oracle harness; queued, `HARVEST_QUEUE.md`). A *malformed* stored program currently
  maps to Syntax error (3) as a hypothesis — exact errnum oracle-pending. CALLIDX in
  SPFUNC/BGFUNC still rides the M3-T6 frame-loop dispatch.
