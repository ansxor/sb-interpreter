# M2 — Graphics (GRP + compositor) (PRD)

> Status: ⬜ · Depends on: M1 · Read `prd/README.md` first. Tasks: `PRD.md` (M2-T*).

## Context / why

SmileBASIC has a layered screen: a backdrop, **graphics pages (GRP)**, BG layers, sprites,
and the console, composited front-to-back by Z priority into the displayed image. M2
builds the GRP drawing model and the **compositor** that produces the faithful RGBA
framebuffer — the same buffer we display and pixel-diff against the oracle. Sprites/BG
(M3) plug into the compositor M2 establishes.

## Goal (Definition of Done)

- GRP pages with correct count/size/format and page selection (GPAGE: display vs draw page).
- All graphics drawing instructions implemented per spec, pixel-exact vs the oracle.
- A compositor that stacks backdrop→GRP→BG(stub ok)→sprite(stub ok)→console by Z and
  emits the framebuffer.
- Golden PNG fixtures harvested from the oracle; deterministic pixel-diff replay passes.

## Scope

**In:** GRP model, drawing primitives, bitmap ops, color model, compositor + Z ordering.
**Out:** sprite/BG *content* (M3) — but the compositor must have their layer slots.

## Reference sources

- Specs: `spec/instructions/*.yaml` `category: Graphics`; `spec/reference/constants.yaml`
  (RGB colors with exact bits).
- Docs: `sb-docs/smilebasic-3/manual/{screen-layout,graphic-instructions,computer-colors-rgb}.md`.
- osb: `graphic.d` (`Graphic2` is the active impl), `petitcomputer.d` (GraphicPage, GL
  page mapping, GRP0-GRP5 per DISPLAY).
- Disassembly: confirm color packing/expansion + primitive rasterization (line/circle/
  fill/paint) exactly — anti-aliasing is OFF on hardware; integer rasterization rules
  matter for pixel-exact diffs. `sb-render::expand5` already encodes the `<<3` rule.

## Architecture decisions
- Draw into an internal GRP surface representation (the on-device color format — likely
  RGBA5551/RGB565-ish; confirm via O-T6/disassembly), then convert to RGBA8888 in the
  compositor. Keep one conversion path (`sb-render`).
- Rasterizers must be deterministic and match SB's pixel coverage exactly (no float AA).

## Tasks

### M2-T1 — GRP page model
- **Files:** `crates/sb-render/src/grp.rs` (new); builtins in `crates/sb-core/src/builtins/graphics.rs` (new).
- **Approach:** model GRP pages (count, 512×512 backing per page — confirm), GPAGE
  (display/draw page selection), GCLS, GCOLOR (current draw color), GPRIO (Z), GCLIP, plus
  color helpers RGB/RGBREAD/GSPOIT. Color values are ARGB8888 to the user (e.g. #WHITE);
  store internally per the device format.
- **Acceptance:** GPAGE/GCLS/GCOLOR specs' tests pass; RGB()/RGBREAD round-trip exact
  (constants in `constants.yaml`).

### M2-T2 — Drawing primitives
- **Files:** `graphics.rs`, `crates/sb-render/src/raster.rs` (new).
- **Approach:** GPSET, GLINE, GBOX, GFILL, GCIRCLE, GTRI, GPAINT (flood fill). Match SB's
  integer rasterization (endpoints inclusive? circle algorithm? paint boundary rule —
  confirm in disassembly/oracle). Coordinate ranges X0-399/Y0-239 (top). → M2-T1
- **Acceptance:** per-primitive golden PNG diffs vs oracle are pixel-exact for a set of
  representative draws (incl. edge cases: 1px lines, degenerate tris, fill seams).

### M2-T3 — Bitmap ops
- **Files:** `graphics.rs`.
- **Approach:** GCOPY (page→page blit, with transforms?), GLOAD (data→page), GSAVE
  (page→data). Respect clip + color-key semantics per spec. → M2-T1
- **Acceptance:** GCOPY/GLOAD/GSAVE round-trip tests + golden diffs.

### M2-T4 — Compositor
- **Files:** `crates/sb-render/src/compositor.rs` (new).
- **Approach:** stack layers by Z (backdrop → GRP(s) → BG → sprite → console) into the
  final `Framebuffer`. Define the layer interface so M3 plugs BG/sprite in. Handle the SB
  Z-depth model (1024 rear … 0 screen … -256 front) and per-DISPLAY GRP mapping. → M2-T2
- **Acceptance:** composited output for a scene (GRP + console) matches a golden PNG; Z
  ordering correct when layers overlap.

### M2-T5 — RE color/pixel math + golden harvest
- **Files:** spec updates; `harness/oracle/framebuffer.py`; `harness/corpus/golden/gfx/`.
- **Approach:** finalize device pixel format + color expansion from the disassembly;
  harvest oracle framebuffers (needs O-T6) for the graphics test set; wire pixel-diff into
  `harness/diff/replay.py`. Raise graphics specs to `disassembled`/`hw_verified`. → M2-T4, O-T6
- **Acceptance:** committed golden PNGs; `replay.py` pixel-diffs them deterministically;
  CI replays without an emulator.

## Milestone verification
```bash
cargo test -p sb-render -p sb-core
cargo run -p sb-platform-native --bin sb -- <a graphics demo>.sb3   # visual
python3 harness/diff/replay.py    # golden gfx count > 0, diffs pass
```

## Risks / open questions
- Exact rasterization rules (circle/paint/triangle fill) are the main fidelity risk — lean
  on oracle golden diffs to pin them.
- Device pixel format + tiling (O-T6) gates pixel-exact comparison; until then, compare in
  RGBA8888 after both sides convert.
