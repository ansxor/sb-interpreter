# M2 — Graphics (GRP + compositor) (PRD)

> Status: ⬜ · Depends on: M1 · Read `prd/README.md` first. Tasks: `PRD.md` (M2-T*).

## Context / why

SmileBASIC has a layered screen: a backdrop, **graphics pages (GRP)**, BG layers, sprites,
and the console, composited front-to-back by Z priority into the displayed image. M2
builds the GRP drawing model and the **compositor** that produces the faithful RGBA
framebuffer — the same buffer we display and pixel-diff against the oracle. Sprites/BG
(M3) plug into the compositor M2 establishes.

## Goal (Definition of Done)

- GRP pages with correct count/size/format and page selection (GPAGE: display vs draw page).
- All graphics drawing instructions implemented per spec, pixel-exact vs the oracle.
- A compositor that stacks backdrop→GRP→BG(stub ok)→sprite(stub ok)→console by Z and
  emits the framebuffer.
- Golden PNG fixtures harvested from the oracle; deterministic pixel-diff replay passes.

## Scope

**In:** GRP model, drawing primitives, bitmap ops, color model, compositor + Z ordering.
**Out:** sprite/BG *content* (M3) — but the compositor must have their layer slots.

## Reference sources

- Specs: `spec/instructions/*.yaml` `category: Graphics`; `spec/reference/constants.yaml`
  (RGB colors with exact bits).
- Docs: `sb-docs/smilebasic-3/manual/{screen-layout,graphic-instructions,computer-colors-rgb}.md`.
- osb: `graphic.d` (`Graphic2` is the active impl), `petitcomputer.d` (GraphicPage, GL
  page mapping, GRP0-GRP5 per DISPLAY).
- Disassembly: confirm color packing/expansion + primitive rasterization (line/circle/
  fill/paint) exactly — anti-aliasing is OFF on hardware; integer rasterization rules
  matter for pixel-exact diffs. `sb-render::expand5` already encodes the `<<3` rule.

## Architecture decisions
- Draw into an internal GRP surface representation (the on-device color format — likely
  RGBA5551/RGB565-ish; confirm via O-T6/disassembly), then convert to RGBA8888 in the
  compositor. Keep one conversion path (`sb-render`).
- Rasterizers must be deterministic and match SB's pixel coverage exactly (no float AA).

## Tasks

### M2-T1 — GRP page model
- **Files:** `crates/sb-render/src/grp.rs` (new); builtins in `crates/sb-core/src/builtins/graphics.rs` (new).
- **Approach:** model GRP pages (count, 512×512 backing per page — confirm), GPAGE
  (display/draw page selection), GCLS, GCOLOR (current draw color), GPRIO (Z), GCLIP, plus
  color helpers RGB/RGBREAD/GSPOIT. Color values are ARGB8888 to the user (e.g. #WHITE);
  store internally per the device format.
- **Acceptance:** GPAGE/GCLS/GCOLOR specs' tests pass; RGB()/RGBREAD round-trip exact
  (constants in `constants.yaml`).
- **Done (M2-T1):** GRP page model in `crates/sb-render/src/grp.rs` — 6× 512×512 RGBA5551
  pages + the command state (display/manip page, draw color, Z prio, display/write clip
  rects) in `GrpState`, with `argb8888_to_rgba5551`/`rgba5551_to_argb8888` (the device 8↔5
  truncation: `<<3` expand, low 3 bits zero — proves the `#WHITE`/GSPOIT round-trip caveat),
  `gcls`/`gspoit`/`gclip_*` methods + 7 unit tests. Builtins in
  `crates/sb-core/src/builtins/graphics.rs` (GPAGE/GCLS/GCOLOR/GPRIO/GCLIP/RGB/RGBREAD/
  GSPOIT) registered in `BUILTIN_NAMES`; the VM owns a `GrpState` and routes them via
  `call_graphics` (like `call_console`). SET/GET forms are chosen by a unified `ret_count`
  (= `wants_value`?1:`out_argc`), matching the disassembled `[r0,#0xc]` return-count check,
  so `GPAGE OUT V,W`, `GCOLOR OUT C`, and `C=GCOLOR()` all route correctly. 9 builtin unit
  tests + 43 conformance cases in `harness/corpus/cases/graphics_grp.yaml` (the hw_verified
  s_t7a/s_t7e/s_c2 value + error cases that PRINT can show). Full gate green.
- **Deferred / queued:** (1) the GSPOIT *post-draw* round-trip spec tests (`GPSET …:GSPOIT`)
  need GPSET → land with M2-T2 (the read path is already proven by the `gcls→gspoit`
  sb-render unit test). (2) Empty `OUT` receivers (`RGBREAD C OUT P,,,` — corpus form) are
  not yet supported: `compile_pop_target` rejects an `ExprKind::Void` target; this is a
  general OUT-feature gap (also affects DTREAD) — implement when DTREAD lands. (3) `GCLIP`
  write-rect rejection → errnum 10 (the page clip-helper reject branch) is not modeled
  (all rects accepted) — the exact reject condition is oracle-pending.

### M2-T2 — Drawing primitives
- **Files:** `graphics.rs`, `crates/sb-render/src/raster.rs` (new).
- **Approach:** GPSET, GLINE, GBOX, GFILL, GCIRCLE, GTRI, GPAINT (flood fill). Match SB's
  integer rasterization (endpoints inclusive? circle algorithm? paint boundary rule —
  confirm in disassembly/oracle). Coordinate ranges X0-399/Y0-239 (top). → M2-T1
- **Acceptance:** per-primitive golden PNG diffs vs oracle are pixel-exact for a set of
  representative draws (incl. edge cases: 1px lines, degenerate tris, fill seams).
- **Done (M2-T2):** all seven primitives — `GPSET`, `GLINE`, `GBOX`, `GFILL`, `GCIRCLE`
  (full circle + arc/sector), `GTRI`, `GPAINT` — implemented as deterministic integer
  rasterizers in `crates/sb-render/src/raster.rs` (`impl GrpState`): Bresenham line (reused
  by GBOX edges + sector radii), midpoint circle, integer edge-function triangle fill (skips
  degenerate/collinear), and a 4-connected stack flood fill (implicit-region vs explicit-
  border forms). Every primitive writes the GPAGE manipulation page in RGBA5551 (same 8→5
  truncation as GCLS) and is bounded to the write-clip ∩ 512×512 page; off-page/clipped plots
  are silently dropped (no error). Builtins in `crates/sb-core/src/builtins/graphics.rs`
  (`gpset`/`gline`/`gbox`/`gfill`/`gcircle`/`gtri`/`gpaint`) enforce the disassembled
  arg-count + return-count guards (→ errnum 4) and the default-color path (current `GCOLOR`),
  registered in `BUILTIN_NAMES` and routed via `call_graphics`. `r<=0` GCIRCLE is a no-op;
  GCIRCLE arg-dispatch covers 3/4 (circle) and 5/6/7 (arc/sector, flag defaults 0). The
  call-shape / arg-count / default-color behavior is **hw_verified** (sb-oracle s_t7b/s_t7c)
  and the specs' inline `tests:` now replay in the conformance gate (`GPSET`/`GLINE`/`GBOX`/
  `GFILL`/`GCIRCLE`/`GTRI`/`GPAINT` added to `IN_SCOPE_GRAPHICS`). Implementing `GPSET` also
  unblocked the three hw_verified `GSPOIT` post-draw round-trip cases (`GPSET …:GSPOIT` →
  -524288 / -460552 / -16752640), so `GSPOIT` moved out of `IN_SCOPE_PARTIAL` and is now
  fully in scope. 11 rasterizer unit tests + 2 builtin-form tests added; full gate green.
- **Deferred / queued (pixel-exact golden diff):** the *exact pixel coverage* (line endpoint
  rule, circle/arc midpoint vs hardware, GCIRCLE angle convention/normalization, GPAINT
  boundary test, triangle edge inclusivity, float→int coordinate rounding) has no scalar
  golden and is faithful-but-unverified. A pixel-exact PNG diff needs a GRP→RGBA8888 render
  path (the compositor, **M2-T4**) plus the diff harness (**M2-T5**), and the reference
  goldens harvested via O-T6 — i.e. the same framebuffer-path work already queued under
  **S-T7b**/**S-T7c** in `HARVEST_QUEUE.md` (now cross-referenced by an M2-T2 entry). The
  acceptance's pixel-diff is the milestone-level DoD realized in M2-T5, not separately
  achievable here (M2-T4 itself depends on M2-T2).

### M2-T3 — Bitmap ops
- **Files:** `graphics.rs`.
- **Approach:** GCOPY (page→page blit, with transforms?), GLOAD (data→page), GSAVE
  (page→data). Respect clip + color-key semantics per spec. → M2-T1
- **Acceptance:** GCOPY/GLOAD/GSAVE round-trip tests + golden diffs.
- **Done (M2-T3):** all three bitmap ops in `crates/sb-render/src/bitmap.rs` (`impl GrpState`:
  `gcopy`/`read_region`/`write_region`/`gsave_word`/`gload_halfword`/`whole_draw_area`) +
  builtins in `crates/sb-core/src/builtins/graphics.rs` (`gcopy`/`gsave`/`gload`), routed via
  `call_graphics`, registered in `BUILTIN_NAMES`, and folded into `IN_SCOPE_GRAPHICS`. GCOPY is
  a snapshot-then-write page→page blit (same-page overlap safe); GSAVE/GLOAD transfer a page
  region to/from a numeric array in two element formats (convert_flag 1 = raw RGBA5551
  halfword, 0 = 32-bit logical ARGB color), the 1-D GSAVE dest auto-expanding. **copy_mode**
  governs transparent (alpha-bit-clear) source pixels: FALSE skips them (destination kept),
  TRUE copies them (destination cleared). The whole-area form (GSAVE/GLOAD with no rect) is
  the write-clip ∩ page = 512×512 = 262144 elements. **All behavior hw_verified** (sb-oracle
  s_t7d_bitmap 2026-06-23): GSPOIT round-trips (both flags → 0xFF00F800), saved element words
  (raw red 0xF801=63489; logical red 0xFFF80000 = -524288 signed / 4294443008 in a Double
  array), whole/sub-rect `LEN` (262144 / 64), GCOPY pixel-move + copy-mode, and every errnum
  (bad count → 4, page ∉ -1..5 → 10, too-small image → 31, string array → 8). Tests: 5
  sb-render unit + 2 sb-core builtin unit + 17 conformance cases
  (`harness/corpus/cases/graphics_bitmap.yaml` round-trips + the gload/gsave/gcopy spec inline
  cases, the latter corrected — a 3-arg whole GLOAD loads 512×512, so it must pair with a
  whole-area GSAVE; a sub-rect GSAVE into a 64-elem array then whole GLOAD raises errnum 31,
  hw_verified). Full gate green (native + wasm).
- **Deferred / queued:** (1) **GRPF (source page -1)** isn't backed — reading it yields
  transparent pixels (the captured-screen content needs the framebuffer model, O-T6;
  HARVEST_QUEUE). (2) **GLOAD form-2 (palette array)** is implemented as the documented
  index→palette recolor (palette entries read as 32-bit logical colors) but its exact palette
  semantics are oracle-pending (HARVEST_QUEUE). (3) The *pixel-exact PNG golden diff* of a blit
  is the milestone DoD realized in M2-T5 (compositor + O-T6 harvest), same as M2-T2.

### M2-T4 — Compositor
- **Files:** `crates/sb-render/src/compositor.rs` (new).
- **Approach:** stack layers by Z (backdrop → GRP(s) → BG → sprite → console) into the
  final `Framebuffer`. Define the layer interface so M3 plugs BG/sprite in. Handle the SB
  Z-depth model (1024 rear … 0 screen … -256 front) and per-DISPLAY GRP mapping. → M2-T2
- **Acceptance:** composited output for a scene (GRP + console) matches a golden PNG; Z
  ordering correct when layers overlap.
- **Done (M2-T4):** compositor in `crates/sb-render/src/compositor.rs`. A `Layer` trait
  (`z()` + `composite(&mut Framebuffer)`) is the M3 plug-in point; M2 ships `GrpLayer` (a GRP
  display page — the visible window is the top-left crop of the 512×512 page, masked by the
  display clip, composited with the device **1-bit alpha key**: opaque pixels expand
  RGBA5551→ARGB8888, clear-alpha pixels show the layers behind) and `ConsoleLayer` (delegates
  to the existing `Console::render`, which already honors per-cell transparency). `compose()`
  fills a fresh framebuffer with the backdrop, then paints layers **rear→front** by Z (stable
  sort, descending Z: `1024` rear … `0` screen … `-256` front, **smaller Z draws in front**);
  an equal-Z tie keeps the caller's slice order, encoding the documented stack
  GRP<BG<sprite<console. `compose_top_screen(grp, console, backdrop)` wires the standard
  scene (GRP display page at its `GPRIO` Z → console front) and is now used by **both**
  platform hosts: `sb-platform-native`'s `sb` window and `sb-platform-wasm`'s
  `render_program` composite the scene (so GRP draws finally appear), replacing the
  console-only render. BG/sprite slots are reserved by the Z model, not hard-coded — M3 adds
  `impl Layer` types. 6 compositor unit tests (opaque-over-backdrop, transparent-shows-
  backdrop, console-in-front-at-equal-Z, smaller-Z-wins regardless of slice order, display-
  clip crop, display-page selection); full gate green (native + wasm). The acceptance's
  **golden PNG** pixel-diff is the milestone DoD realized in **M2-T5** (it needs the O-T6
  composite-framebuffer harvest); Z ordering + layer compositing are verified deterministically
  here. Oracle-pending (queued in `HARVEST_QUEUE.md`): per-layer **default Z** + exact equal-Z
  tie-break, the `BACKCOLOR`→backdrop mapping/default, and the **partial-alpha** composite rule
  (M2 uses only the device 1-bit key) — all need the O-T6 composite capture.

## Risks / open questions
- Exact rasterization rules (circle/paint/triangle fill) are the main fidelity risk — lean
  on oracle golden diffs to pin them.
- Device pixel format + tiling (O-T6) gates pixel-exact comparison; until then, compare in
  RGBA8888 after both sides convert.
