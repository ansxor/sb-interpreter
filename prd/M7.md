# M7 — Hardening (PRD)

> Status: ⬜ · Depends on: M2-M6 · Read `prd/README.md` first. Tasks: `PRD.md` (M7-T*).

## Context / why

By M7 the interpreter is feature-complete. This milestone pushes it from "works" to
"faithful": broad differential fuzzing against the oracle, raising spec confidence to
`hw_verified` across the set, and nailing the fiddly numeric/string corners (float
formatting, integer overflow, exact error strings). The deliverable is a measurably
high-fidelity interpreter with a large committed regression corpus.

## Goal (Definition of Done)

- Sustained fuzzing campaign with findings minimized + promoted into the committed corpus.
- The large majority of specs at `disassembled`/`hw_verified`, tracked by the coverage
  report.
- Error messages/numbers, float formatting, and overflow/precision corners match the real
  binary; performance is acceptable on native + wasm.

## Reference sources
- Everything: specs, disassembly (error strings ≈ `0x1E965C`; float-format + math
  routines), the oracle (`harness/`), and the coverage report (`sb-spec-coverage`).

## Tasks

### M7-T1 — Fuzzing campaign
- **Files:** `harness/fuzz/generator.py` (full grammar), `harness/diff/run.py`,
  `harness/harvest/harvest.py`.
- **Approach:** flesh out the seeded, grammar-aware generator (spec-signature-driven so
  calls are well-typed). Run differential vs the oracle; minimize each divergence to a
  small program; **promote** it into `harness/corpus/` (or a `spec/tests/` overlay) as a
  permanent seeded regression. → O-T8
- **Acceptance:** a documented fuzzing run; every found divergence either fixed in sb-core
  or captured as a known-divergence case; promoted cases replay deterministically in CI.

### M7-T2 — hw_verified push
- **Files:** `spec/tests/*`, spec `sources`/`confidence` fields; `harness/harvest`.
- **Approach:** harvest oracle `expect:` values across the spec corpus; bump confidence.
  Prioritize math, strings, graphics, sound. Track progress via the coverage report. → O-T8
- **Acceptance:** coverage report shows a large `hw_verified` share (set a concrete target,
  e.g. ≥80% of implemented instructions); CI replays the frozen expects.
- **Progress (2026-06-23, run 1):** First oracle-harvest pass against real SB 3.6.0 (Azahar).
  - **classify → `hw_verified`** (Mathematics now **27/27**). The headline inf→1 / NaN→2
    paths — previously the last unverified piece — were produced via `double` overflow:
    `EXP(1000)`→+inf, `EXP(1000)-EXP(1000)`→NaN, so `CLASSIFY(EXP(1000))=1`,
    `CLASSIFY(-EXP(1000))=1`, `CLASSIFY(EXP(1000)-EXP(1000))=2` (confirms the @0x20c3e0
    helper map; HARVEST_QUEUE S-T1b resolved). 3 inline cases added; sb-core passes them.
  - **Various-input category (13 instructions) frozen as committed fixtures.** `harvest.py
    --category "Various kinds of input"` drove the oracle: **33/33 cases confirmed, 0
    mismatch, 0 capture failures** → `spec/tests/{accel,brepeat,button,gyroa,gyrosync,gyrov,
    micdata,micsave,micstart,micstop,stick,stickex,touch}.yaml`. Hand-added the two OUT-form
    baselines `harvest.py` skips (`STICK`/`STICKEX OUT X,Y`→`0,0`, via `run_case` progstr).
    These error-guards + neutral baselines are now regression-locked + CI-replayed.
  - **Coverage 51.2% → 51.6% hw_verified** (127→128); 1192→1230 frozen cases.
  - **Structural ceiling (why ≥80% is not reachable purely in-loop):** the input/mic/motion/
    wireless family stays top-level `disassembled` on purpose (consistent with M6-T5's
    treatment of this exact hardware-stub family) — only their headless-observable contract
    (errnum guards + neutral 0 baselines) is oracle-confirmable; their LIVE hardware path
    (button bitmask, analog stick, accel/gyro values, mic samples, the post-`XON` sensor
    path, multi-console MP) is permanently un-harvestable headless. And the bulk of the
    remaining `disassembled` set is **Graphics (18) / Sprites (19) / BG (23) / Sound (18)**
    whose headline behavior needs framebuffer/audio goldens (O-T6 composite / O-T7 audio,
    both deferred) or real hardware. So the in-loop hw_verified gain is bounded by the small
    software/value-deterministic remainder; reaching ≥80% requires the deferred media +
    hardware + multi-console harvest, not more value/errnum batches.
  - **Path forward (future runs):** keep harvesting one value/errnum-deterministic category
    per run — Wireless, Screen control, the Files/Control/Data-op stragglers — freezing each
    as `spec/tests/*` regression fixtures (the % may plateau but the deterministic corpus
    grows). The maintainer closes the ≥80% gap via the media/hardware goldens.
- **Progress (2026-06-23, run 2):** Screen-control GET-form harvest against real SB 3.6.0
  (Azahar). The two Screen-control commands with a **scalar GET form** were promoted to
  `hw_verified` (top-level): **BACKCOLOR** and **DISPLAY** (Screen control 2/7 → 4/7).
  - **BACKCOLOR round trip — alpha strip discovered.** `BACKCOLOR c : ?BACKCOLOR()` returns
    the stored color **masked to 24 bits** (`GET = stored & &H00FFFFFF`): RGB(64,128,128)
    (&HFF408080) → 4227200 (&H408080), RGB(16,32,64) → 1056832 (no channel swap), and an
    arbitrary high byte is dropped too (`&H7F112233` → &H112233, `-1` → &HFFFFFF). The spec
    had wrongly claimed a full round-trip; corrected. **Fixed sb-core**: `backcolor()` SET now
    masks `& 0x00FF_FFFF` (it stored/returned the value verbatim). 8 inline value goldens +
    a `backcolor_get_strips_alpha` unit test.
  - **DISPLAY GET** returns the active output-screen id: `?DISPLAY()` → 0 (Upper default),
    `XSCREEN 2:DISPLAY 1:?DISPLAY()` → 1. sb-core already matched; added 3 inline value
    goldens. ACLS/VISIBLE/XSCREEN have **no scalar GET form** (headline = screen state, only
    errnum guards harvestable — already done), so they honestly stay `disassembled`.
  - **Coverage 51.6% → 52.4% hw_verified** (128→130); 1230→1241 frozen cases. Harvest TSVs in
    `harness/harvest/out/screen_get*.tsv`.
- **Progress (2026-06-23, run 3):** Wireless-communication category (8 commands) error-guard
  harvest against real SB 3.6.0 (Azahar) via `harvest.py --category "Wireless communication"`.
  **19/19 error-guard cases confirmed, 0 mismatch** → `spec/tests/mp{end,get,name,recv,send,
  set,start,stat}.yaml` (`mpname.too_many_args` re-run by hand after one transient capture
  failure). The cases cover every headless-reachable validation guard: argcount errnum 4
  (all 8), MPSET/MPSTART range errnum 10, MPSEND/MPSTART type errnum 8, MPSEND empty-string
  errnum 4. Added/extended `hw_verified` source refs on mp{end,set,start,stat,send}.
  - **Discovery — the MP-restriction flag @0x305612 is ZERO in a normal RUN context.** The
    range/type guards sit AFTER the flag check in every handler; their firing as errnum 10/8
    (never 52) PROVES the flag is unset, so a plain program reaches them rather than bouncing
    on errnum 52 "Incompatible statement". This **resolves HARVEST_QUEUE S-T13a open question
    (a)** (a valid `MPSTART` proceeds toward a real connection — hence must NOT be harvested
    headless; it would hang). errnum 52 itself stays oracle-pending (no headless way to set
    the flag).
  - **Top-level confidence stays `disassembled` for all 8** (the structural ceiling): only the
    pre-network validation contract is headless-observable; the LIVE messaging path (MPSTAT
    0/1 connection status, MPNAME$ names, MPGET slot values, MPSEND/MPRECV delivery + errnum
    41/42, errnum 52) needs ≥2 real wireless peers a single Azahar can't form. So the
    hw_verified instruction count is unchanged (still 130) but the deterministic corpus grows.
  - **1241 → 1260 frozen cases.** Harvest TSV in `harness/harvest/out/batch-8.tsv`.
- **Progress (2026-06-24, run 4):** S-T4f Data-ops stragglers — **KEY** and **OPTION** promoted
  to `hw_verified` (Data operations 9/14 → 11/14; coverage 52.4% → **53.2%** hw_verified,
  130 → 132). Two genuinely-unharvested queue items resolved against real SB 3.6.0 (Azahar):
  - **KEY() undocumented function form** reads back the bound string: `KEY 3,"HI":?KEY(3)`="HI"
    (also "CLS"/"HELLO"); function-form guards KEY(6)/KEY(0) → errnum 10, KEY(1,2) → errnum 4.
    Frozen in `spec/tests/key.yaml` (6 cases); sb-core already matches.
  - **OPTION STRICT/DEFINT/TOOL**: STRICT + undeclared `B=2` → errnum 15 errline 2; OPTION TOOL
    compiles cleanly (recognized feature, no error); unknown feature → errnum 3. **OPTION DEFINT
    flips the suffix-less numeric default Real→Integer**, and the real→int conversion **truncates
    toward zero** (A=3.7→3, A=-3.7→-3, A=2.5→2, A=-2.5→-2; A#=3.7 stays 3.7). Decisively, WITHOUT
    any OPTION the suffix-less default is **Real** (A=3.7→3.7, `DIM A[3]:A[0]=3.7`→3.7) — resolving
    the long-standing "suffix-less Real-vs-Int" open question. Frozen in `spec/tests/option.yaml`.
  - **Harness fix (the big one): `spec/tests/*.yaml` overlays were NOT being replayed.** The
    conformance loader's `CaseFile` read a `cases:` key, but every overlay (and `harvest.py`
    output) uses `tests:` — serde silently dropped them to zero cases. So all 23 prior overlay
    files (accel/button/stick/mp*/…, 65 cases) were dead weight, never exercised. Added
    `#[serde(alias = "tests")]` so the gate now replays them — directly fulfilling M7-T2's
    acceptance ("CI replays the frozen expects"), which had not actually been happening. Proven
    via a canary (`corpus_and_overlay_cases_pass` now reports 342 curated cases, all green).
  - **sb-core impl gap found (queued, M1-T4 — NOT this slice):** sb-core types unsuffixed `DIM`
    arrays as Int (should be Real) and treats OPTION DEFINT as a no-op (flag parsed, never
    consumed). So the 4 DEFINT real→int coercion cases are recorded hw_verified in option.yaml's
    sources but NOT frozen as conformance cases yet; corrected the misleading `bytecode.rs`
    comments and queued the fix in `HARVEST_QUEUE.md`.
- **Progress (2026-06-24, run 5):** **PRINT** promoted `disassembled` → **hw_verified**
  (Console input/output 9/12 → 10/12; coverage 53.2% → **53.6%** hw_verified, 132 → 133). The
  number formatter was already hw_verified (M7-T4 run); this run harvested the remaining
  headline behavior — separators, integer formatting, and the newline rule — against real SB
  3.6.0 (Azahar) by reconstructing the rendered console row with `CHR$(CHKCHR(I,0))` up to CSRX
  and reading CSRX/CSRY back (so the goldens are the REAL cells, not inferred values):
  - **Integer = %d, no leading/trailing space:** `PRINT 42`="42", `PRINT -7`="-7", `PRINT 0`="0",
    `PRINT 100` → CSRX=3.
  - **`;` concatenates no-gap:** `PRINT "A";"B"`="AB", `PRINT 5;"X"`="5X", `PRINT 1;2;3`="123".
  - **`,` tabs to the next STRICTLY-greater multiple of TABSTEP** (`?TABSTEP`=4): `PRINT "A","B"`
    ="A   B" (CSRX after `PRINT "A",`=4); decisively, **a cursor already ON a boundary advances a
    full step** — `PRINT "ABCD","Z"`="ABCD    Z" (CSRX after `PRINT "ABCD",`=8, not 4). This
    settled the open edge case; sb-core's `(cur_x/step+1)*step` already matched.
  - **Trailing `;`/`,` suppresses the closing newline** (`LOCATE 0,3:PRINT "X";` → CSRY=3; same
    for `,`); no trailing separator emits one newline (`PRINT "X"` → CSRY=4, CSRX=0); a **bare
    `PRINT` emits exactly one line break** (CSRY=4).
  - 10 cases frozen in `spec/tests/print.yaml` (replayed by the overlay gate); 4 new `vm.rs`
    unit tests (boundary-advance, num;str, trailing-comma, bare-PRINT blank line). sb-core
    matched every harvested value with **no code change** — PRINT's separator/newline lowering
    was already correct. Harvest TSVs in `harness/harvest/out/print_out{,2}.tsv`.

### M7-T4 — Float formatting
- **Files:** `crates/sb-core/src/builtins/{math,string}.rs` (STR$/PRINT/FORMAT$/VAL).
- **Approach:** reverse-engineer SB's exact double→string algorithm from the disassembly
  (the `otya_test` `VAL(STR$(RNDF))==0.836095` shows ~6 sig figs; pin rounding, exponent
  threshold, trailing-zero rules). Apply to PRINT/STR$/FORMAT$. → disassembly
- **Acceptance:** a broad table of float→string cases matches the oracle exactly; the
  M1 "best-effort" formatter specs flip to `hw_verified`.
- **Progress (DONE 2026-06-23):** The disassembly shows SB uses TWO different doube→string
  formatters, both literal C `sprintf` calls: **STR$** = `%g` at 6 sig figs (handler
  @0x1eb2a8, fmt "%g" @0x1eb4a8), **PRINT** = `%.8f` then trailing-zero/bare-dot trim
  (handler @0x180a50, fmt "%.8f" @0x180b0c, trim loop @0x180a8c) — fixed, never exponential.
  `builtins::format_g` already matched `%g` (verified exact against a 2000-case bit-exact
  `%.6g` sweep); the bug was PRINT reusing it. Added `format_print_number`/`format_fixed8`
  (`%.8f`+trim) and routed `format_print_item` through it. Both keep signed zero ("-0").
  hw_verified PRINT values harvested via a `PRINT`+`CHKCHR` console read-back (e.g. PRINT
  12345678.0="12345678" where STR$="1.23457e+07"; PRINT 0.00001="0.00001"; PRINT 1/3=
  "0.33333333"; PRINT -0.0="-0"). Updated 24 math-spec test cases (they expected the STR$
  `%g` value under a bare `PRINT`), added broad %g/%.8f conformance tables to str.yaml &
  print.yaml + Rust unit tables. (E-notation source literals like `1E100` are a separate
  parser gap — already queued, covered here by a unit-test instead.)

### M7-T5 — Overflow/precision + performance
- **Files:** core hot paths; `vm.rs`.
- **Approach:** verify i32 overflow/wrap + int↔double promotion corners (MAX/MIN overflow,
  shifts, DIV/MOD sign rules) against the oracle. Profile the VM; optimize dispatch/value
  representation without breaking determinism. Check wasm perf.
- **Acceptance:** overflow/precision cases match oracle; benchmark a heavy program at
  acceptable fps native + wasm; no determinism regressions.

## Risks / open questions
- Some behaviors may be emulator-faithful but not hardware-faithful; cross-check
  suspicious cases against real-hardware run logs where possible.
- Float-format RE is the highest-effort numeric task; budget accordingly.
