# M7 — Hardening (PRD)

> Status: ⬜ · Depends on: M2-M6 · Read `prd/README.md` first. Tasks: `PRD.md` (M7-T*).

## Context / why

By M7 the interpreter is feature-complete. This milestone pushes it from "works" to
"faithful": broad differential fuzzing against the oracle, raising spec confidence to
`hw_verified` across the set, and nailing the fiddly numeric/string corners (float
formatting, integer overflow, exact error strings). The deliverable is a measurably
high-fidelity interpreter with a large committed regression corpus.

## Goal (Definition of Done)

- Sustained fuzzing campaign with findings minimized + promoted into the committed corpus.
- The large majority of specs at `disassembled`/`hw_verified`, tracked by the coverage
  report.
- Error messages/numbers, float formatting, and overflow/precision corners match the real
  binary; performance is acceptable on native + wasm.

## Reference sources
- Everything: specs, disassembly (error strings ≈ `0x1E965C`; float-format + math
  routines), the oracle (`harness/`), and the coverage report (`sb-spec-coverage`).

## Tasks

### M7-T1 — Fuzzing campaign
- **Files:** `harness/fuzz/generator.py` (full grammar), `harness/diff/run.py`,
  `harness/harvest/harvest.py`.
- **Approach:** flesh out the seeded, grammar-aware generator (spec-signature-driven so
  calls are well-typed). Run differential vs the oracle; minimize each divergence to a
  small program; **promote** it into `harness/corpus/` (or a `spec/tests/` overlay) as a
  permanent seeded regression. → O-T8
- **Acceptance:** a documented fuzzing run; every found divergence either fixed in sb-core
  or captured as a known-divergence case; promoted cases replay deterministically in CI.

### M7-T2 — hw_verified push
- **Files:** `spec/tests/*`, spec `sources`/`confidence` fields; `harness/harvest`.
- **Approach:** harvest oracle `expect:` values across the spec corpus; bump confidence.
  Prioritize math, strings, graphics, sound. Track progress via the coverage report. → O-T8
- **Acceptance:** coverage report shows a large `hw_verified` share (set a concrete target,
  e.g. ≥80% of implemented instructions); CI replays the frozen expects.
- **Progress (2026-06-23, run 1):** First oracle-harvest pass against real SB 3.6.0 (Azahar).
  - **classify → `hw_verified`** (Mathematics now **27/27**). The headline inf→1 / NaN→2
    paths — previously the last unverified piece — were produced via `double` overflow:
    `EXP(1000)`→+inf, `EXP(1000)-EXP(1000)`→NaN, so `CLASSIFY(EXP(1000))=1`,
    `CLASSIFY(-EXP(1000))=1`, `CLASSIFY(EXP(1000)-EXP(1000))=2` (confirms the @0x20c3e0
    helper map; HARVEST_QUEUE S-T1b resolved). 3 inline cases added; sb-core passes them.
  - **Various-input category (13 instructions) frozen as committed fixtures.** `harvest.py
    --category "Various kinds of input"` drove the oracle: **33/33 cases confirmed, 0
    mismatch, 0 capture failures** → `spec/tests/{accel,brepeat,button,gyroa,gyrosync,gyrov,
    micdata,micsave,micstart,micstop,stick,stickex,touch}.yaml`. Hand-added the two OUT-form
    baselines `harvest.py` skips (`STICK`/`STICKEX OUT X,Y`→`0,0`, via `run_case` progstr).
    These error-guards + neutral baselines are now regression-locked + CI-replayed.
  - **Coverage 51.2% → 51.6% hw_verified** (127→128); 1192→1230 frozen cases.
  - **Structural ceiling (why ≥80% is not reachable purely in-loop):** the input/mic/motion/
    wireless family stays top-level `disassembled` on purpose (consistent with M6-T5's
    treatment of this exact hardware-stub family) — only their headless-observable contract
    (errnum guards + neutral 0 baselines) is oracle-confirmable; their LIVE hardware path
    (button bitmask, analog stick, accel/gyro values, mic samples, the post-`XON` sensor
    path, multi-console MP) is permanently un-harvestable headless. And the bulk of the
    remaining `disassembled` set is **Graphics (18) / Sprites (19) / BG (23) / Sound (18)**
    whose headline behavior needs framebuffer/audio goldens (O-T6 composite / O-T7 audio,
    both deferred) or real hardware. So the in-loop hw_verified gain is bounded by the small
    software/value-deterministic remainder; reaching ≥80% requires the deferred media +
    hardware + multi-console harvest, not more value/errnum batches.
  - **Path forward (future runs):** keep harvesting one value/errnum-deterministic category
    per run — Wireless, Screen control, the Files/Control/Data-op stragglers — freezing each
    as `spec/tests/*` regression fixtures (the % may plateau but the deterministic corpus
    grows). The maintainer closes the ≥80% gap via the media/hardware goldens.

### M7-T3 — Exact error strings
- **Files:** `crates/sb-core/src/error.rs`, spec `errors`.
- **Approach:** reconcile each error's message + number against the strings in the
  disassembly (`0x1E965C`+) and oracle behavior (capitalization/punctuation). → O-T5
- **Acceptance:** error messages match the binary byte-for-byte where SB surfaces them;
  errnum assignments verified.

### M7-T4 — Float formatting
- **Files:** `crates/sb-core/src/builtins/{math,string}.rs` (STR$/PRINT/FORMAT$/VAL).
- **Approach:** reverse-engineer SB's exact double→string algorithm from the disassembly
  (the `otya_test` `VAL(STR$(RNDF))==0.836095` shows ~6 sig figs; pin rounding, exponent
  threshold, trailing-zero rules). Apply to PRINT/STR$/FORMAT$. → disassembly
- **Acceptance:** a broad table of float→string cases matches the oracle exactly; the
  M1 "best-effort" formatter specs flip to `hw_verified`.
- **Progress (DONE 2026-06-23):** The disassembly shows SB uses TWO different doube→string
  formatters, both literal C `sprintf` calls: **STR$** = `%g` at 6 sig figs (handler
  @0x1eb2a8, fmt "%g" @0x1eb4a8), **PRINT** = `%.8f` then trailing-zero/bare-dot trim
  (handler @0x180a50, fmt "%.8f" @0x180b0c, trim loop @0x180a8c) — fixed, never exponential.
  `builtins::format_g` already matched `%g` (verified exact against a 2000-case bit-exact
  `%.6g` sweep); the bug was PRINT reusing it. Added `format_print_number`/`format_fixed8`
  (`%.8f`+trim) and routed `format_print_item` through it. Both keep signed zero ("-0").
  hw_verified PRINT values harvested via a `PRINT`+`CHKCHR` console read-back (e.g. PRINT
  12345678.0="12345678" where STR$="1.23457e+07"; PRINT 0.00001="0.00001"; PRINT 1/3=
  "0.33333333"; PRINT -0.0="-0"). Updated 24 math-spec test cases (they expected the STR$
  `%g` value under a bare `PRINT`), added broad %g/%.8f conformance tables to str.yaml &
  print.yaml + Rust unit tables. (E-notation source literals like `1E100` are a separate
  parser gap — already queued, covered here by a unit-test instead.)

### M7-T5 — Overflow/precision + performance
- **Files:** core hot paths; `vm.rs`.
- **Approach:** verify i32 overflow/wrap + int↔double promotion corners (MAX/MIN overflow,
  shifts, DIV/MOD sign rules) against the oracle. Profile the VM; optimize dispatch/value
  representation without breaking determinism. Check wasm perf.
- **Acceptance:** overflow/precision cases match oracle; benchmark a heavy program at
  acceptable fps native + wasm; no determinism regressions.

## Risks / open questions
- Some behaviors may be emulator-faithful but not hardware-faithful; cross-check
  suspicious cases against real-hardware run logs where possible.
- Float-format RE is the highest-effort numeric task; budget accordingly.
