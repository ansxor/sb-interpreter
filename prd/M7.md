# M7 — Hardening (PRD)

> Status: ⬜ · Depends on: M2-M6 · Read `prd/README.md` first. Tasks: `PRD.md` (M7-T*).

## Context / why

By M7 the interpreter is feature-complete. This milestone pushes it from "works" to
"faithful": broad differential fuzzing against the oracle, raising spec confidence to
`hw_verified` across the set, and nailing the fiddly numeric/string corners (float
formatting, integer overflow, exact error strings). The deliverable is a measurably
high-fidelity interpreter with a large committed regression corpus.

## Goal (Definition of Done)

- Sustained fuzzing campaign with findings minimized + promoted into the committed corpus.
- The large majority of specs at `disassembled`/`hw_verified`, tracked by the coverage
  report.
- Error messages/numbers, float formatting, and overflow/precision corners match the real
  binary; performance is acceptable on native + wasm.

## Reference sources
- Everything: specs, disassembly (error strings ≈ `0x1E965C`; float-format + math
  routines), the oracle (`harness/`), and the coverage report (`sb-spec-coverage`).

## Tasks

### M7-T1 — Fuzzing campaign
- **Files:** `harness/fuzz/generator.py` (full grammar), `harness/diff/run.py`,
  `harness/harvest/harvest.py`.
- **Approach:** flesh out the seeded, grammar-aware generator (spec-signature-driven so
  calls are well-typed). Run differential vs the oracle; minimize each divergence to a
  small program; **promote** it into `harness/corpus/` (or a `spec/tests/` overlay) as a
  permanent seeded regression. → O-T8
- **Acceptance:** a documented fuzzing run; every found divergence either fixed in sb-core
  or captured as a known-divergence case; promoted cases replay deterministically in CI.

### M7-T5 — Overflow/precision + performance
- **Files:** core hot paths; `vm.rs`.
- **Approach:** verify i32 overflow/wrap + int↔double promotion corners (MAX/MIN overflow,
  shifts, DIV/MOD sign rules) against the oracle. Profile the VM; optimize dispatch/value
  representation without breaking determinism. Check wasm perf.
- **Acceptance:** overflow/precision cases match oracle; benchmark a heavy program at
  acceptable fps native + wasm; no determinism regressions.

- **Progress (2026-06-24):** Harvested the i32 overflow / precision corners against real
  SB 3.6.0 (sb-oracle, 5 batches) and fixed sb-core to match — the headline finding is that
  SmileBASIC `+`/`-`/`*` are **type-aware**, not uniformly wrapping:
  - **Overflow PROMOTES vs WRAPS by operand static type.** A statically-Integer operand
    (`%` var, integer literal, bit-op result) wraps mod 2³² (`A%=2000000000:A%*2 →
    -294967296`); a Real/Number-typed operand (suffix-less / `#` / Real literal / `/`-result
    / constant fold) promotes Int→Double on overflow (`A=2147483647:A+1 → 2147483648.0`,
    `2147483647+1 → 2.14748e+09`). Implemented via a new compiler `is_real_typed` →
    `Op::OperatePromote` / `vm::operate_promote` (i64-compute-then-narrow-or-promote) plus
    the same promotion in the parser constant folder (`int_arith`). Previously sb-core
    wrapped everything (correct only for `%`-typed).
  - **Double→`%`-store Overflow.** Storing a Double outside `[i32::MIN, i32::MAX]` into a
    `%` variable or array element raises **Overflow (errnum 9)** — the guard fires on the
    value before truncation (`A%=2147483647.9` → 9; `A%=2147483647.0` → 2147483647). New
    `Value::to_int_store`, wired into `coerce_to_suffix(Int)`, `ElemRef::Int` store, and
    `pop_array`. The arithmetic-operand truncation `to_int` (AND/DIV/MOD/shift) still
    saturates (unchanged).
  - **Shift count is not masked mod 32.** `<<` with count `<0` or `≥32` → 0; `>>` is
    arithmetic (sign-extending), count `<0` → 0, count `≥32` saturates to the sign bit. Was
    masking via Rust `wrapping_shl`/`wrapping_shr` (`1<<32` wrongly gave 1).
  - **DIV/MOD** integer wrap on `INT_MIN/-1` (`-2147483648 DIV -1 → -2147483648`, `MOD → 0`)
    confirmed already correct; frozen.
  - **Tests:** `harness/corpus/cases/overflow.yaml` (24 hw_verified cases) + `value.rs` /
    `vm.rs` / `compiler.rs` / `parser.rs` unit tests; updated `m1_conformance.sb3` and the
    two old fixtures that encoded the wrong (uniform-wrap) behavior. Determinism guard
    (`heavy_program_is_deterministic`) + a release throughput probe: **~6.9M loop-iter/s
    native** (≈45M VM ops/s) — `sb-core` is platform-free so wasm runs the same dispatch,
    comfortably above interactive rates; no profiler hotspot warranted a rewrite.
  - **Left `[ ]` — queued in `HARVEST_QUEUE.md`:** MIXED `%`+suffix-less operand typing
    (sb-core promotes; unverified), builtin/sysvar return-type promotion (conservatively
    wraps), INC/DEC + FOR-counter promotion (still wrap), unsuffixed-array element promotion
    (gated on the M1-T4 unsuffixed-array-Real default), NaN→`%` store, and a dedicated fps
    benchmark harness. The core verified overflow/precision cases all match the oracle.
