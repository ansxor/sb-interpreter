# M4 — Input & timing (PRD)

> Status: ⬜ · Depends on: M1 (parallel with M2/M3) · Read `prd/README.md`. Tasks: `PRD.md` (M4-T*).

## Context / why

Interactive programs need buttons, touch, sticks, and a frame clock. SmileBASIC runs at
60 fps and exposes input + timing through builtins and system variables. Faithful timing
(VSYNC/WAIT/MAINCNT) is also what makes animation (M3) and audio (M5) line up. This
milestone can proceed in parallel with M2/M3 once M1's window/loop exists.

## Goal (Definition of Done)

- BUTTON/STICK/STICKEX/TOUCH/KEY/INKEY$ return faithful state.
- A 60 fps main loop with VSYNC/WAIT and a correct MAINCNT.
- Display/hardware config (XSCREEN/DISPLAY/VISIBLE/WIDTH, HARDWARE) behaves per spec.
- Host input (native keyboard/gamepad; wasm keyboard/gamepad) maps to SB input state.
- Deterministic tests via a scripted/headless input timeline (no live device needed).

## Reference sources
- Specs: `category` Various input / Screen control in `spec/instructions/`;
  `spec/reference/sysvars.yaml` (MAINCNT, HARDWARE), `constants.yaml` (button bits
  #UP…#ZR with exact values).
- Docs: `sb-docs/smilebasic-3/manual/` input/screen topics.
- osb: `petitcomputer.d` (input handling, SDL mapping, Hardware enum, XSCREEN/DISPLAY,
  VSYNC/frame loop), `systemvariable.d` (MAINCNT).

## Architecture decisions
- `sb-core` exposes an **input state struct** the platform layer fills each frame; builtins
  read from it. Keep devices out of `sb-core` (wasm-safe). Provide a **scripted input
  source** for deterministic tests (a timeline of frame→buttons/touch).
- A single frame clock drives MAINCNT, VSYNC, animation (M3), and audio scheduling (M5).

## Tasks

### M4-T1 — Buttons / sticks
- **Files:** `crates/sb-core/src/input.rs` (new); `crates/sb-core/src/builtins/input.rs` (new).
- **Approach:** BUTTON (with the press/repeat/release modes per spec), STICK, STICKEX,
  BREPEAT (auto-repeat config). Button bitmask from `constants.yaml` (#A=&H10 …).
- **Acceptance:** BUTTON modes + BREPEAT match spec under a scripted input timeline;
  bit values exact.
- **Done (2026-06-23):** `crates/sb-core/src/input.rs` (`InputState`: per-frame held/
  pressed-raw/pressed-with-repeat/released masks + the two analog sticks + per-button
  `BREPEAT` repeat config, driven by `advance_frame` — the scripted timeline) and
  `crates/sb-core/src/builtins/input.rs` (`BUTTON`/`STICK`/`STICKEX`/`BREPEAT`). Wired into
  the VM via `call_input` over a new `Vm::input`/`input_mut` (mirrors the graphics/sprite/BG
  routing); registered in `BUILTIN_NAMES` + the conformance `IN_SCOPE_INPUT` list. Bit values
  exact (`#ZR`=2048/b11, `#ZL`=4096/b12 per the hw_verified constants; b10 unused/masked);
  the four feature masks + the BREPEAT repeat re-fire are covered by `input.rs` scripted-
  timeline unit tests and VM integration tests; the inline spec error/baseline cases replay in
  the conformance gate. Live magnitudes / default-repeat / errnum-52 wireless paths are not
  headless-harvestable — queued in `HARVEST_QUEUE.md`.

### M4-T2 — Touch + keyboard
- **Files:** `input.rs`, `builtins/input.rs`.
- **Approach:** TOUCH (x,y + touch state on the lower screen), KEY (on-screen keyboard),
  INKEY$. → M4-T1
- **Acceptance:** TOUCH returns correct coords/state from scripted input; INKEY$ drains a
  key queue per spec.

### M4-T3 — Frame timing
- **Files:** `crates/sb-core/src/clock.rs` (new); VSYNC/WAIT in `builtins`; MAINCNT sysvar.
- **Approach:** model frames; VSYNC(n) waits n frames, WAIT(n) similar per spec; MAINCNT =
  frames since launch. The loop in `sb-platform-*` ticks the clock at 60 fps; headless mode
  advances the clock deterministically. → M1-T12
- **Acceptance:** VSYNC/WAIT advance MAINCNT by the right counts; deterministic in headless
  mode; native loop holds ~60 fps.
- **Done (2026-06-23):** New `crates/sb-core/src/clock.rs` (`FrameClock`) owns the single
  free-running frame counter (`[0x315ec0]`, == MAINCNT) plus the per-program last-VSYNC anchor
  (`[0x315ee8]`), with `wait`/`vsync`/`tick`/`maincnt` and the `FPS`/`FRAME_DURATION` (1/60 s,
  `core::time`) pacing constants. The VM's bare `frame_count`/`last_vsync` fields collapsed
  into this clock; `call_timing` (WAIT/VSYNC) now delegates to it (same documented call shape:
  statement-only → errnum 4, default 1, negative → 0). **MAINCNT** wired as a read-only sysvar:
  the compiler resolves a bare `MAINCNT` read to new `Op::PushMaincnt` (reserved, resolves
  before user vars) and rejects assignment as a compile-time Syntax error (errnum 3) via the
  shared `is_readonly_sysvar` (ERR* + MAINCNT) — **hw_verified** `MAINCNT=5`→errnum 3 errline 1
  (sb-oracle 2026-06-23). New `Vm::maincnt()` + `Vm::tick_frame()` (the `swi 0xa` per-VBlank
  heartbeat: advance the counter + step sprite/BG animation). The native `sb` window now keeps
  the VM and runs a real **60 fps loop** (`ControlFlow::WaitUntil(FRAME_DURATION)` +
  `step_frame` + re-blit), so animations set up before the program returned keep playing.
  Tests: 6 `clock` unit tests (anchor/jitter/zero-count), VM MAINCNT/VSYNC/WAIT integration
  tests + read-only rejection, native `frame_loop_ticks_maincnt_and_animation`. The live
  program-driven VSYNC loop (VM *yielding* per frame to the host) remains a later milestone;
  MAINCNT boot-value/monotonicity across RUN/CONT stays queued (`HARVEST_QUEUE.md`).

### M4-T4 — Display config
- **Files:** `builtins/screen.rs` (new); `crates/sb-render` layout.
- **Approach:** XSCREEN (screen mode / layer config), DISPLAY (top/bottom target),
  VISIBLE (per-layer visibility), WIDTH (console width 50/80?), HARDWARE sysvar (1=new3DS).
- **Acceptance:** XSCREEN/DISPLAY change which GRP/console maps where (verified via M2
  compositor + golden diff); HARDWARE returns the configured model.
- **Progress (done):** `builtins/screen.rs` adds a VM-owned `ScreenConfig` (XSCREEN mode,
  DISPLAY target, per-screen VISIBLE flags, HARDWARE model) routed by `Vm::call_screen`.
  XSCREEN/DISPLAY/VISIBLE enforce the disassembled arg-shape (→ errnum 4) and range
  (→ errnum 10) guards — all hw_verified via sb-oracle batch s_t11d, now folded into the
  conformance gate (`IN_SCOPE_SCREEN`) and the VM unit tests. DISPLAY's GET form round-trips
  the selected screen; its range check tracks the active XSCREEN mode (DISPLAY 1 needs
  XSCREEN 2/3). VISIBLE booleanizes the four layer flags and the compositor
  (`compose_top_screen(..., LayerVisibility)`) drops a hidden layer group — verified by
  golden-style pixel tests in `sb-render`. HARDWARE reads as a read-only bare-name sysvar
  (1 = new3DS); assigning to it is a compile-time Syntax error (errnum 3).
- **Deferred (queued in `HARVEST_QUEUE.md`):** the reimplementation renders only the Upper
  screen, so DISPLAY 1 → Touch-screen output, the XSCREEN sprite/BG split *across* screens,
  Touch-screen VISIBLE rendering, and the DIRECT-mode errnum-43 guards are not yet realized
  (no second-screen framebuffer / no composite oracle golden — O-T6).

### M4-T5 — Host input mapping
- **Files:** `crates/sb-platform-native/src/main.rs`, `crates/sb-platform-wasm/src/lib.rs`.
- **Approach:** map native keyboard/gamepad and wasm keyboard/gamepad events into the
  `sb-core` input state each frame. Provide a default keymap; document it. → M4-T1, M4-T2
- **Acceptance:** pressing keys/buttons drives BUTTON/TOUCH/STICK in a running program on
  both native and wasm.
